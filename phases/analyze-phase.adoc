= ONTAP Foreign LUN Import（FLI）分析階段
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
分析階段旨在確定在進入詳細的移轉規劃和執行階段之前必須修復哪些問題。此階段的主要目標是確保每台主機及其所依賴的完整 SAN 堆疊（作業系統、磁碟區管理器、HBA、多路徑、交換器韌體以及目標 ONTAP 版本）都能遷移到支援的移轉後組態。具體做法是：將每台主機的當前狀態與 NetApp 互通性矩陣（IMT）中支援的組合進行比較、然後為每台主機定義一個清晰的目標組態（作業系統層級、MPIO/ALUA 行為、HBA 驅動程式 / 韌體以及任何所需的主機工具）、並產生一份差距分析報告、列出達到該目標所需的具體變更。

安排並審查差距分析，以避免因補救措施而導致的變更，這些變更可能會影響應用程式相容性和停機計劃。在許多環境中，某些變更（例如 OS 修補程式或 HBA 更新）在遷移視窗之前完成更為安全。然而，多路徑堆疊變更通常會延遲到切換或遷移之後，因為更改 MPIO 行為可能會影響目前陣列的支援性，並使回滾複雜化。

對於 iSCSI FLI ，分析階段也應涵蓋 iSCSI 特有的就緒情況：

* 確認主機的 iSCSI 多重路徑設計在完全移轉後仍受支援
* 確保外部陣列符合 iSCSI 後端使用資格，因為後端傳輸差異可能會改變互通性行為。




== ONTAP FLI 分析階段任務

分析階段以主機為中心，但也應驗證完整的端對端支援鏈（主機到 Fabric 到 ONTAP 再返回），並確認 FLI 後端互通性位置。

[cols="1,2"]
|===
| 元件 | 工作 


| 主機  a| 
* 將目前主機組態與 IMT 支援的組態進行比較，並確定移轉後的目標組態：
+
** OS 層級
** 多重路徑方法
** HBA 型號、驅動程式和韌體
** 所需的主機工具


* 建立包含以下必要資訊的每個主機差距分析：
+
** Hotfix 與修補程式
** 作業系統更新
** HBA 驅動程式和韌體升級
** 多重路徑變更
** 列出任何必要的公用程式






| Fabric（FC/iSCSI 網路）  a| 
* 在端對端視圖中驗證交換器和 Fabric 支援（型號和韌體），如果目標 ONTAP 和主機組合不支援 Fabric，則記錄所需的升級。




| NetApp 儲存陣列（目的地）  a| 
* 確認目標 ONTAP 平台和版本與預期的 FLI 工作流程相容，包括任何適用的 AFF 啟用預期，並記錄影響目標設計的任何先決條件。




| 外部儲存陣列（來源）  a| 
* 驗證來源陣列是否在 FLI 後端互通性的支援範圍內。如果 ONTAP 9.9.1 及更高版本的 IMT 中未列出互通性，請使用 SAN LUN Migrate App 來協助確定其是否受支援。


|===


== 在 ONTAP FLI 分析階段使用 IMT

互通性矩陣工具 (IMT) 是一個組態儲存庫，它提供了一種引導方式，幫助使用者縮小範圍，找到 NetApp 產品與第三方元件互通的端對端支援組態。IMT 包含支援的組態和經過認證的組態：

* 支援的組態由 NetApp 認證。
* 經過認證的組態是由第三方機構驗證其能夠與 NetApp 元件配合使用。


.檢查 IMT 是否包含以下內容：
* 在工作表中記錄 IMT 動作：在規劃工作表中記錄 IMT 推薦的主機和交換器軟體、修補程式和升級。
* 先從廣泛入手，然後逐步細化：先確定靜態標準（ONTAP 版本、協定、HA/叢集模式），然後根據現場勘測結果新增主機作業系統、磁碟區管理器和 HBA 詳細資訊。
* 避免過度篩選：如果您的搜尋過於具體而沒有傳回任何結果，請放寬篩選條件、檢閱多個有效組態，然後選擇最佳支援的相符項目。
* 規範 HBA 識別碼：HBA 可能顯示為 OEM 零件編號，在輸入 IMT 之前進行交叉引用。
* 檢查每台主機：對遷移範圍內的每台主機執行 IMT 支援性檢查。
* 進行兩項互通性檢查：
+
** 後端 FLI 支援：確認來源陣列後端案例是否受支援。
** 端對端移轉後支援:確認對 ONTAP、主機作業系統、多重路徑、HBA 和交換器架構組合的支援。


* 實際 IMT 工作流程：
+
** 搜尋來源陣列型號，然後選擇「Foreign LUN Import（FLI）後端互通性」，並選擇 ONTAP 平台和 ONTAP 版本。
** 為了驗證主機相容性，請使用「使用 ONTAP SAN 主機建置端對端視圖」來驗證作業系統、多路徑和 HBA。
** 對於交換器相容性，請使用 SAN 主機視圖中的「Build end-to-end view for SAN-Switch」來驗證 Fabric 和交換器支援。




.若外部陣列未在 IMT（ONTAP 9.9.1 及更新版本）中列出
從 ONTAP 9.9.1 開始，如果外部陣列未在 IMT 中列為支援，您可以使用 SAN LUN 遷移工具來協助判斷該陣列是否可能支援 FLI。該應用程式可從 NetApp 支援網站下載，並在具有對來源陣列 LUN 區塊存取（FC 或 iSCSI）的 Linux 主機上執行。結果將由工程團隊審核。從外部陣列匯入可以進行現場認證。對於 iSCSI FLI，從外部陣列匯入始終需要進行現場認證。



== ONTAP FLI 主機變更與復原策略

分析階段應明確決定哪些內容需要更改以及何時更改，因為主機變更可能會引入風險，並且還會影響對目前儲存的支援。如果主機堆疊需要升級才能獲得 ONTAP 支援，這一點尤其重要。

* 主機更新（修補程式、HBA 韌體和驅動程式）如果可能提前完成會更安全，但仍需要與維護視窗和應用程式團隊協調。
* 多路徑堆疊（MPIO、ALUA 和第三方多路徑）的變更可能會影響目前陣列的支援性，因此通常會延遲到移轉事件甚至切換之後。
* 如果在停機期間必須恢復到原始儲存設備，延遲 MPIO 重新配置可以簡化復原過程。




== ONTAP FLI 差距分析

差距分析總結了目前環境與 NetApp 建議目標組態之間的差異，並列出為達到受支援的移轉後狀態所需的升級或補救措施。

每個主機的完整目標組態通常包括：

* 作業系統組態層級和所需的修補程式或 Hotfix
* 多路徑方法和所需設定（MPIO 和 ALUA 行為
* HBA 和 CNA 型號、驅動程式和韌體版本
* 與您的環境相關的主機公用程式和工具需求（如果適用）


應盡快審查差距分析，因為補救措施可能需要停機時間，並且可能會影響應用程式支援需求。
