= ONTAP Foreign LUN Import（FLI）規劃與準備階段
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
計劃和準備階段將發現和分析結果轉化為可執行的遷移計劃，並確認環境已為遷移視窗做好準備。專案的大部分工作通常都集中在這個階段，因為在此階段需要最終確定切換方案（線上或離線）、驗證端對端連線，並透過一次或多次測試遷移來驗證工作流程，然後再移至正式作業環境。

對於 iSCSI FLI，此階段也必須驗證 iSCSI 後端設計約束。ONTAP iSCSI 軟體啟動器使用 IC LIF，需要明確的目標入口網站 IP 和目標 IQN，無需探索工作階段，並且設計為在定義的規模限制內運行（每個節點的匯入數、每個 LIF/節點的工作階段數、每個 LIF 的啟動器數、最大匯入大小）。



== ONTAP FLI 規劃與準備階段結果

* 使用主機差距分析最終確定補救計畫，包括移轉前進行的變更，以及切換視窗期間進行的變更，例如 MPIO 變更。
* 建置來源到目的地 LUN 對應、來源 LUN 識別碼和序號到目的地 ONTAP LUN 名稱和路徑，包括復原對應。
* 使用 IC LIF 可達性和全網狀會話拓撲結構（其中每個 HA 節點與每個目標入口網站建立會話）驗證 ONTAP 到外部陣列的後端連接。這是可靠發現外部 LUN 的先決條件。
* 執行一次或多次測試移轉，以確認組態正確性並收集處理量樣本，從而協助估算正式作業持續時間和人員配置工作量。




== ONTAP FLI 規劃與準備階段任務

以下是各元件的 ONTAP FLI 計畫和準備工作。

[cols="1,2"]
|===
| 元件 | 工作 


| 主機  a| 
* 使用差距分析最終確定主機修復計畫，包括變更應該在移轉前、切換期間還是移轉後進行。
* 根據停機容忍度和工作負載關鍵性選擇線上或離線移轉模式；相應地調整修復時間。
* 驗證並記錄在移轉期間要使用的預期使用案例（標準 LUN、叢集磁碟、RDM/VMFS 資料存放區等）的主機端重新探索和補救步驟。




| 網路 / 網路  a| 
* FC 環境：備份現有區域集，並規劃切換和移轉後清理所需的區域變更。
* iSCSI 後端環境：驗證 ONTAP IC LIF 與外部陣列目標入口網站 IP 之間的端對端 IP 可達性（路由 / VLAN / 防火牆，包括 TCP/3260），並確保冗餘。
* 不要僅為了存取外部目標而在不同子網路上新增 IC LIF，當其他子網路上已有使用中的 IC LIF 時（視為不受支援的規劃）。




| NetApp 儲存陣列（目標 ONTAP）  a| 
* 確認 ONTAP 叢集已準備好進行 iSCSI FLI（支援的版本 / ECV），並且移轉後將刪除僅用於移轉的 iSCSI 後端組態（這對升級 / 還原準備工作很重要）。
* 識別並確認 LUN 擁有節點和 HA 合作夥伴上的 IC LIF，並規劃恢復能力（通常每個節點在不同連接埠上有兩個 IC LIF）。
* 建立軟體 iSCSI 啟動器（每個 IC LIF 一個啟動器）並建立全網狀工作階段（每個 HA 節點對應每個外部目標入口網站和 IQN），並驗證工作階段是否正常。
* 使用 storage disk show 指令在 ONTAP 中探索外部 LUN、驗證路徑健全狀況和基本內容（包括 512B 需求），然後將外部 LUN 標記為外部。
* 使用 -foreign-disk 參數配置具有大小同位的測試目的地 LUN，使用計劃的前端傳輸協定（FC 或 iSCSI）來對應它，並執行至少一次測試移轉以驗證端對端工作流程並擷取處理量。
* 確保計劃符合 iSCSI FLI 限制（無探索工作階段、無 MCS、僅限 ERL0、匯入和工作階段限制、以及匯入大小上限）。




| 外部儲存陣列（來源）  a| 
* 收集目標入口網站 IP 和目標 IQN，工作階段是明確建立的，不使用探索工作階段。
* 為 ONTAP 啟動器建立廠商端存取物件（主機/主機群組等效物），並將小型測試 LUN（通常為 ~1 GB）僅對應到 ONTAP 啟動器以進行資格認證和測試。
* 驗證外部目標是否能看到 ONTAP 啟動器登入和工作階段，以及是否已對應正確的 LUN 以進行匯入。
* 確保在切換過程中，生產主機不會同時呈現相同的來源 LUN，以避免雙重呈現。
* 在正式匯入之前，使用支援的鑑定工具和方法完成外部陣列鑑定。


|===
